Using the Extend BRD skill

Draft, revise, review, and verify client-facing Business Requirements Documents for Workday Extend / Built-on-Workday apps — filled into the standard Invisors template.  ·  skill: extend-brd

Start it

Type /extend-brd, or just ask in plain language — it triggers on BRD / requirements-doc work even if you don't say "BRD" (e.g. "turn these notes into a BRD," "document this app's processes," or naming a client/app). Give it one of three inputs: a scoping doc / meeting notes, a live working conversation, or the app's files (model/, presentation/, orchestration/) to reverse-document.

What to expect

  • It asks first. It assesses your inputs and asks a tight batch of gap questions up front — answer them; it will not invent the "why."
  • It drafts into the real template and delivers a .docx (cover, Lato fonts, TOC, footer preserved).
  • It flags limits. Designs that brush a documented Extend platform limit get called out as you go.
  • Unknowns become Open:/TBD — that's correct behavior, not a gap in quality.

The gap questions it will ask

These are the things files and notes almost never carry — have answers ready:

  • Business problem + volume / scale
  • Security role owners & user counts
  • Data retention for the records it creates
  • Language set (defaults to US English only)
  • Whether mobile access is needed

Four things you can ask for

  • Draft — produce a new BRD from your inputs.
  • Revise — hand it an existing BRD; it edits in place as tracked changes (author "Claude") to accept/reject.
  • Review (ask for it) — "review this BRD / build the RAID log": surfaces Open Questions + RAID (Risks, Actions, Issues, Decisions) — measured in the doc, blunt in chat.
  • Verify (ask for it) — "verify this BRD / check for hallucinations": traces every claim to a source and walks you through anything unverified.

Get the best results

  • Hand over the actual app files when reverse-documenting — grounded beats guessed.
  • Answer the gap questions — that's where accuracy comes from.
  • Run the verification pass before sending to a client — it catches fabricated numbers and claims.
  • Don't paste the whole conversation into the client doc — it writes forward-looking requirements, not a transcript.

What it won't do (by design)

  • Invent volumes, retention periods, roles, or limits it can't source.
  • Assert a platform limit unless it's verified — otherwise it asks you to confirm.
  • Produce internal design docs or ROM / scoping estimates (different deliverable).
  • Rebuild the template's cover or TOC — it edits the real template.

About the output

  • A .docx in the Invisors template — open it in Word once and the TOC page numbers recalculate (stale numbers beforehand are expected).
  • The section formerly called "Processes" is titled Features.
  • Revisions arrive as tracked changes; fresh drafts arrive as plain content.

Maintainers only: the Extend-limits reference is refreshed by a Playwright crawler (scripts/scrape_extend_limits.py) and the installable package is built with scripts/build_package.py — neither is needed for normal use. See TESTING.md to test a new build.
